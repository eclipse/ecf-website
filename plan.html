 ECF Plan

Pollinate's goal is to build an Eclipse-based IDE and toolset that leverages the open source Apache Beehive application framework. There is a tight focus on Beehive related functionality making everything else that Pollinate needs secondary. Pollinate will reuse what it can from the Web Tools Project.
 

 ECF Project Dependencies

Naturally, Pollinate is closely tied to Apache Beehive, and will track its builds closely.

Also, Pollinate will draw on the fruits of the Eclipse Web Tools Platform Project (WTP) as it can. Since WTP is just getting under way, it is too soon to say how Pollinate will relate to it, although it is likely that Pollinate will in some fashion be a consumer of WTP artifacts. If a WTP piece of functionality is needed by Pollinate, but not yet complete, then we may contribute energy to WTP to get that functionality completed.

The JDT Core team is feverishly at work on J2SE 1.5 features (see Bug 36938) including annotation support (JSR 175), on which we depend.

Beehive also specifies some web services, so JSR 181 will be of interest in the future.
  

 Milestone 1 Plan - September 30, 2004

Milestone 1 is  devoted to implementing the "create, edit, and deploy a web application" use case. The earliest implementations of that use case will be very primitive, consisting of a new project wizard, a Java editor that does some annotation validation, the builder and other infrastructure to support that, and a script for building and deploying the web application. The idea is to make some basic progress on implementation while getting project management issues like setting up a build process out of the way.

The initial focus is on NetUI PageFlows, leaving Controls and Web Services for future milestones.

  • Simple pollinate feature including beehive and tomcat plugins
  • No extra install... no need for user to install tomcat or beehive
  • Simple wizard to create a new webapp project
  • Simple wizard to create page flow
  • Simple command to launch tomcat on webapp project in Eclipse
  • Pollinate feature installable via update manager

The user must not need to install Beehive or Tomcat to use Pollinate. Both will be included as plugins as part of the Pollinate deliverable. We want this usable "right out of the box" using the Eclipse update manager.
 Milestone 2 Plan - November 11, 2004
Milestone 2 has been rescheduled from mid-December to November 11 so that Pollinate can be presented together with Beehive at ApacheCon. Obviously that will mean cutting back on what we were planning to accomplish in M2, but it seemed like a reasonable request.

For Milestone 2, we are focusing on editors and views for the code that we build and execute. We will have to build specialized parts such as a page flow editor, but perhaps we can pull in other parts, such as a JSP editor, from WTP or others.

  • java page flow editor - navigation only (Carl)
  • continue work on runtime launching (Alex)
  • move build and update site to proper Eclipse.org download servers (Dan)
  • moved to latest version of Beehive (Carl)
  • [mostly complete] automated nightly build - build script in place but not quite fully automated yet (Dan)
  • [deferred to M3] enhancing the various Pollinate wizards (Lee)
  • [deferred to M3] runtime console view (if we have time)

 

 Milestone 3 Plan - end of Feb 2005 just before EclipseCon
Milestone 3 is focused on finishing page flow support and adding some support for Beehive controls.
We intend to ship right before EclipseCon and demo what we have accomplished there.

Some things we want to tackle based upon Pollinate's goal include:

  • java page flow editor - visual editing, missing annotation support, and bug fixes (Carl)
  • enhancing the various Pollinate wizards (Lee)
  • support files named *.jpf (currently java page flow files have to be named *.java) (Carl)
  • runtime support for beehive controls & PetStore example - assuming that the JDT supports runtime annotations (see bug 36938)
  • explore using WTP server tooling in Pollinate (Alex)
  • Allow debugging of web apps from Pollinate - part of WTP server tooling (Alex)
  • move to new version of Tomcat (Alex)
  • move to new version of Beehive (Carl)
  • review WTP JSP editor (if there is one) or other existing contributions (Lee)
  • web-services runtime support (Lee)
  • other beehive specific editors such as those for visually editing controls
  • JUnit tests
  • how do we get rid of our reliance on the "tools.jar" ?
  • new deployment UI similar to java launching and external tools launching dialogs?
  • you can also use Bugzilla to List Outstanding Bugs and Issues for M3

In the future once Pollinate has implemented the basic infrastructure and editors for Beehive, I think Pollinate should explore the new capabilities that Beehive provide, i.e. VISUALLY (!) create, compose an application and deploy J2EE components without a lot of "plumbing". An added value of Pollinate project can be to explore different visualization techniques in addition to standard IDE tasks like project management, code editing, one touch deployment process, etc. Using example of Action view page for JPF, I would say the ultimate goal will be to make it less "read-only" but more agile and convenient for manipulating different artifacts properties for the Flow Pages. The key is finding a way to display a complex set of objects and their relationships in a simple and intuitive way. These are areas we need to discuss and explore.
