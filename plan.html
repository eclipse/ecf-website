 ECF Feature Development Plan

ECF has at least two critical target user groups: plugin developers and user teams. Plugin developers will use the framework to build their applications above ECF, while user teams will use the ECF applications for doing their daily individual and team work.

These two groups can be broken down further into sub-groups. See Plugin Developers and User Teams for descriptions of these sub-groups.

In order to create both user team added value and plugin developer added value we propose 2 'feature vectors' for ECF applications: 'synchronous application features' and 'core framework features'.These vectors will allow the ECF team to:

  1. Define an initial sequence of ECF features to design/implement for each milestone release

  2. Plan the milestone-based implementation/delivery of features

  3. Communicate this plan publicly to the ECF plugin developer community

  4. Receive continuous feedback about feature importance from the community

  5. Receive feedback about desired featured sequencing from the community

  6. Coordinate ECF committers on tasks and dependencies

  7. Allow the community to participate in an open development process

  8. Allow us on the ECF team to easily coordinate and build upon each other's ideas

Below are listed a number of features for each of the two categories of 'synchronous', and 'core framework features'.

 ECF Plugin Developers
There are four major plugin developer groups relevant for ECF
  1. providers: Plugin developers that implement existing or new communications protocols and supply these protocols as provider modules underneath the ECF APIs. The primary value created by these developers is one of interoperability. That is, with multiple provider implementations, other plugin developers and users can allow their Eclipse/ECF-based applications interoperate with other systems (e.g. IM systems, file sharing systems, web-based APIs, etc).

  2. component builders: Plugin developers who implement generic component-level features for communications features (e.g. file sharing, chat, shared whiteboard, team RSS, etc). These developers provide application-level features that can be combined into full ECF-based applications. Their primary value is to provide reusable communications components to application designers.

  3. tool integrators: Plugin developers who integrate existing and/or new tools with ECF to create some new application-level value for an individual or work team. For example, the integration of ECF with EMF/SDO creates an ability for applications to create and share models among multiple users. Their primary value comes from taking features and technologies and combining them into new, more-useful applications and features.

  4. ui builders/integrators: Plugin developers who create and/or integrate UIs for user access to features exposed via other plugins. Their primary value is in creating improved user interfaces, by design or by integration.
